# 👥 Critérios de aceitação e Permanência

## Objetivo
Os critérios de aceitação e permanência servem para garantir uma equipe coêsa, participativa e colaborativa.

## Critérios para aceitação de um novo participante
### 1. Ter comprometimento mínimo

Para ser um novo integrante do grupo BugHunters espera-se comprometimento com o grupo e as tarefas. Sempre buscando bons resultados, atualizando o grupo de novas informações e realizando as entregas das tarefas.

### 2. Ter boa comunicação 

Para ser um novo integrante do grupo BugHunters espera-se que a pessoa tenha uma comunicação com o grupo, sempre informando caso esteja com alguma dificuldade nas tarefas, caso haja alguma adversidade que o impeça de realizar a tarefa, caso não consiga comparecer nas reuniões etc.

### 3. Uso das ferramentas

Todos os integrantes do grupo devem ter conhecimento mínimo das ferramentas utilizadas no projeto (Java, SceneBuilder, Github, Jira e MySQL). Caso não tenha nenhum conhecimento de alguma ou todas as ferramentas, deve ao menos estar disposto e ter dedicação para aprender o quanto antes.

### 4. Responsabilidade com as tarefas

No grupo BugHunters todos esperamos que cada integrante seja responsável e se atente às entregas; tanto para realizar entregas com a melhor qualidade possível, quanto para sempre conseguir entregar no prazo.

## Critérios para a permanência dos integrantes
### 1. Seguir o padrão de commits

Cada participante ao terminar de trabalhar no projeto deve fazer o commit e seguir os padrões estabelecidos no documento <a href ="./padrão-de-commit.md"> Padrão de commits</a>
> **Caso de banimento:** Aquele que realizar mais de 5 commits fora do padrão, deverá se retirar do grupo.

### 2. Seguir o fluxo de branches corretamente

Sempre que um participante for trabalhar em alguma task nova, deve-se criar uma nova branch. Nunca devem trabalhar diretamente na branch Main. Para mais informações, consultar o documento <a href ="./estratégia-de-branch.md"> Estratégia de branches</a>
> **Caso de banimento:** Aquele que fizer mal uso das branches, fizer commit diretamente na Main ou não criar uma nova feature para as tasks mais de 5x, deverá se retirar do grupo.

### 3. Comunicar sobre andamento da tarefa

Foi acordado com o grupo de haver uma comunicação diária através do aplicativo de mensagens WhatsApp. Sempre devendo comunicar o andamento da tarefa que esta fazendo.
> **Caso de banimento:** Aquele que ficar mais de 2 semanas sem mandar alguma atualização sobre a tarefa e sem comunicar um motivo para tal, deverá se retirar do grupo.

### 4. Fazer curso de git e github

Foi acordado com o grupo que todos devem fazer um curso de git\github para melhor utilização da ferramenta e seguimento dos padrões de commit e fluxo de branches.
> **Caso de banimento:** Aquele que fizer mal uso das ferramentas GIT e GitHub por falta de conhecimento, deverá se retirar do grupo.

### 5. Participar das reuniões

As reuniões presenciais ficaram definidas para todas as quartas-feiras em horário de aula (Aulas dedicadas à elaboração do API). Todos os participantes devem participar ou comunicar préviamente caso não consigam comparecer.
> **Caso de banimento:** Aquele que faltar à mais de 3 reuniões sem um aviso prévio ou uma justificativa válida, deverá se retirar do grupo.

### 6. Bom relacionamento interpessoal

Afim de manter uma boa relação do grupo espera-se de cada integrante respeito e educação com os outros colegas. Qualquer espécie de assédio não será tolerado.
> **Caso de banimento:** Aquele que assediar moralmente, virtualmente ou sexualmente algum colega, deverá se retirar do grupo.
